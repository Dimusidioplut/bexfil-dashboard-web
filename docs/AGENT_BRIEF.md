# Agent Brief

## Главный принцип

Собирай минимально работающий дашборд под требования заказчика. Не добавляй новые сущности, экраны и сценарии без прямой необходимости.

## Решение по фреймворку

Фреймворк сейчас не менять.

Остаёмся на `Streamlit` для MVP, потому что:
- он уже подключён;
- позволяет быстро собрать рабочую версию;
- задача сейчас не в смене стека, а в упрощении интерфейса и приведении его к требованиям.

Смена фреймворка допускается только если `Streamlit` объективно блокирует реализацию базового сценария. Пока такого основания нет.

## Что такое MVP в этом проекте

Нужна простая, понятная и рабочая система без лишней аналитики.

В MVP входят только:
- `План / факт` по доходам;
- `План / факт` по расходам;
- `Упущенные договоры`, только если для них есть подтверждённые данные;
- выбор периода;
- месячный обзор;
- недельный оперативный срез;
- tooltip с точными значениями.

## Что НЕ делать

Не делать:
- лишние сущности “на будущее”;
- сложные промежуточные модели, если они не нужны для текущих экранов;
- отдельные большие разделы про рынок, enrichment, производство, стратегию собственника, если это не используется прямо сейчас;
- графики и таблицы “для красоты”, если они не помогают принять решение;
- интерпретацию отсутствующих данных как нуля.

## Что должно быть в интерфейсе

Интерфейс должен быть максимально простым.

Оставить только 3 экрана:
- `План / Факт`
- `Оперативка`
- `Потери и возможности`

Структура:
- `План / Факт`:
  - доходы;
  - расходы;
  - упущенные договоры;
- `Оперативка`:
  - недельный план;
  - недельный факт, если он подтвержден и загружен;
- `Потери и возможности`:
  - только при наличии подтвержденных данных.

Если данных по потерям нет:
- явно показывать пустое состояние;
- не подменять его “кандидатами” из тендеров.

## Правила отображения

- `План` — синий.
- `Факт` — зелёный.
- Неблагоприятное отклонение — красный.
- Благоприятное отклонение — жёлтый.

Интерпретация:
- для расходов:
  - факт выше плана — плохо, красный;
  - факт ниже плана — хорошо, жёлтый;
- для доходов:
  - факт ниже плана — плохо, красный;
  - факт выше плана — хорошо, жёлтый.

## Какие данные использовать

Использовать только то, что реально есть:

- `Данные для ДДС.xlsx`
  - месячный план;
  - недельный план;
- ручной факт из `finance_actuals.csv`, если он заполнен;
- `lost_contracts.csv`, если он заполнен.

Тендерные выгрузки и база компаний не тащить в MVP-интерфейс, если это не нужно для текущего экрана `План / Факт`.

## Приоритет реализации

1. Упростить текущий интерфейс.
2. Убрать лишние разделы и лишние блоки.
3. Оставить только те метрики, которые понятны заказчику сразу.
4. Сделать корректный `план / факт` без выдуманных данных.
5. Явно показать, где факта пока нет.

## Критерий готовности

Система считается готовой для текущего этапа, если:
- заказчик с первого взгляда понимает, где доходы, где расходы, где отклонение;
- можно выбрать период;
- можно посмотреть месяц и неделю;
- интерфейс не перегружен;
- нет лишних сущностей и “аналитики ради аналитики”;
- всё строится на подтверждённых данных.
