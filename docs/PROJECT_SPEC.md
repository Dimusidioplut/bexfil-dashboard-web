# Product Spec

## Цель проекта

Собрать управленческий дашборд Bexfil, который опирается только на подтвержденные данные, не меняет исходные документы и явно показывает границы доступности данных.

Базовый принцип:
- `data_bexfil` — только raw-источники;
- дашборд не выдумывает факт, потери и pipeline, если этого нет в raw-данных;
- все недостающие сущности заводятся отдельными нормализованными таблицами в `dashboard/data_workspace/manual`.

## Главная функция

Основная функция дашборда — наглядное сравнение плановых и фактических данных по:
- доходам;
- расходам;
- упущенным договорам.

Сравнение доступно за любой выбранный период, но только в пределах реально загруженных данных.

Если в источниках отсутствует факт, дашборд обязан явно показывать, что:
- план есть;
- факт не загружен;
- график не должен интерпретировать отсутствие факта как ноль.

## Решение по цветовой логике

Финализированное правило для v1:
- `план` — синий;
- `факт` — зеленый;
- `красный` — неблагоприятное отклонение;
- `желтый` — благоприятное отклонение.

Интерпретация:
- для `расходов`:
  - `факт > план` — красный;
  - `факт < план` — желтый;
- для `доходов`:
  - `факт < план` — красный;
  - `факт > план` — желтый.

## Реально доступные данные в текущем проекте

### Финансы

Источник:
- `data_bexfil/Данные для ДДС.xlsx`

Что есть:
- месячный план: `апрель-июнь 2026`;
- недельный план: `2026-04-16` — `2026-06-25`;
- частично заполненные недельные значения внутри того же файла.

Что важно:
- в raw-данных нет полноценного финансового факта за всю историю;
- нельзя честно построить ретроспективный `plan/fact` за 2022-2025 только по `data_bexfil`.

### Тендеры

Источники:
- `data_bexfil/Контур.Закупки_20.04.2026*.xlsx`

Что есть:
- история закупок за `2022-2025`;
- заказчики, ИНН, регионы, площадки, НМЦ, даты публикаций и дедлайны;
- материал для рыночной аналитики и построения кандидатов в продажи.

Что важно:
- в raw-данных нет признака фактического участия Bexfil;
- нет признака проигрыша;
- нет причин потерь.

### Компании

Источник:
- `data_bexfil/Бэкс - 06.02.2026 база на согласование.xlsx`

Что есть:
- справочник компаний;
- ИНН;
- отрасль;
- регион;
- адрес;
- выручка 2024.

### Упущенные договоры, pipeline, производство

В raw-данных отсутствуют:
- явные `lost_contracts`;
- статусы по возможным сделкам;
- причины потерь;
- заключенные договоры как отдельный реестр;
- производственная готовность к отгрузке;
- фактическая себестоимость.

Для этих блоков dашборд должен использовать отдельные ручные или внешние таблицы.

## Экранная структура v1

### 1. План / Факт

Назначение:
- наглядное сравнение плана и факта по ключевым управленческим показателям.

Содержимое:
- доходы;
- расходы;
- упущенные договоры.

Поведение:
- пользователь выбирает период;
- экран показывает только подтвержденные данные;
- если факта нет, это явно обозначается;
- по наведению видны план, факт и отклонение.

### 2. Оперативка

Назначение:
- недельная рабочая панель.

Содержимое:
- недельный план;
- недельный факт, если он подтвержден и загружен.

Поведение:
- пользователь выбирает недельный период;
- если факта нет, показывается только план и предупреждение;
- экран должен быть максимально компактным и быстро читаемым.

### 3. Потери и возможности

Назначение:
- отдельный контур для подтвержденных потерь и возможностей.

Содержимое:
- возможные сделки;
- подтвержденные упущенные договоры;
- причины потерь;
- статусы взаимодействия.

Важно:
- экран показывается как рабочий только при наличии подтвержденных данных;
- если данных нет, показывается пустое состояние и требуемая схема импорта.

## Обязательные ручные таблицы для v1.1+

Нужно завести отдельные файлы:
- `finance_actuals.csv`
- `opportunities.csv`
- `lost_contracts.csv`
- `production_status.csv`

Они являются частью проекта, но не заменяют raw-источники.

## Правила реализации

- не смешивать `план`, `факт`, `тендерный рынок` и `pipeline` в одном слое;
- не трактовать пустые данные как нулевые значения;
- явно показывать источник и покрытие каждого виджета;
- строить UI так, чтобы он был читаем и на телефоне, и на компьютере;
- использовать графики только там, где они помогают, а не усложняют чтение.
