# Test Server Playbook

## Назначение

Сервер `beta.bigcom.ru` используется как общий тестовый хост для нескольких параллельных проектов. Проекты не должны мешать друг другу ни кодом, ни URL, ни данными.

## Базовый принцип

Каждый проект получает свой `slug` и живёт изолированно:

- URL: `https://beta.bigcom.ru/<slug>/`
- статика: `/var/www/beta.bigcom.ru/<slug>/`
- описание проекта: `/srv/projects/registry/<slug>.md`
- секреты проекта: отдельный файл вне git с правами `600`

## Что уже заведено

- текущий проект: `bexfil_dashboard`
- URL: `https://beta.bigcom.ru/bexfil_dashboard/`
- текущая статика: `/var/www/beta.bigcom.ru/bexfil_dashboard`
- текущий nginx site: `/etc/nginx/sites-available/beta.bigcom.ru.conf`

## Стандарт для новых проектов

Для каждого нового проекта:

1. Выбрать короткий `slug`, только ASCII и `_` или `-`.
2. Создать подпуть `https://beta.bigcom.ru/<slug>/`.
3. Положить deploy-артефакт в `/var/www/beta.bigcom.ru/<slug>/`.
4. Добавить запись в `/srv/projects/registry/<slug>.md`.
5. Если нужен backend, дать проекту отдельный сервис и отдельную БД.

## Что писать в реестр проекта

Минимум:

- название проекта
- `slug`
- URL
- git-репозиторий
- локальный путь репозитория
- путь деплоя на сервере
- где лежат данные
- есть ли PostgreSQL
- какие systemd/nginx сущности связаны с проектом

## Правило по данным и БД

- одна PostgreSQL база или как минимум отдельная схема на проект
- отдельный пользователь БД на проект
- пароли и DSN не хранить в git
- в реестре хранить только место, где лежит секрет, а не сам секрет

## Правило по nginx

- один общий site для `beta.bigcom.ru`
- отдельный `location /<slug>/ { ... }` на каждый проект
- не использовать root-редиректы между проектами
- корень `/` держать пустым или сделать простой index-перечень проектов позже

## Минимальный onboarding для агента

Перед любой работой агент должен посмотреть:

1. `/srv/projects/registry/PROJECTS.md`
2. `/srv/projects/registry/<slug>.md`
3. локальный `AGENT_CONTEXT.md` в репозитории проекта

Если проект новый, агент сначала должен завести запись в серверном реестре, а потом деплоить.
